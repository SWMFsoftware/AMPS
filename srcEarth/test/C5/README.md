# C5 — Dipole mesh-interpolation convergence

C5 validates the **Mode3D mesh-backed magnetic-field path** for the Earth SEP / geospace backward tracing products.

C1 with `--mode3d-field-eval ANALYTIC` checks the clean analytical dipole problem without mesh interpolation. C5 asks the next question:

```text
If the same dipole field is stored on the Mode3D AMR mesh and interpolated during backtracing, does the cutoff solution approach the analytical Størmer result as the mesh is refined?
```

This test is important because production Mode3D runs often use magnetic and electric fields stored on a mesh. A pusher can pass the analytical dipole test while the mesh-backed path still fails because of field materialization, AMR resolution, interpolation, ghost-cell filling, MPI materialization, or thread-safety issues. C5 isolates those mesh-specific errors.

## What C5 tests

C5 runs a centered, aligned dipole with `E_FIELD OFF`, vertical arrival directions, and a 9000 km spherical shell. The analytical reference is the vertical Størmer cutoff:

```text
Rc = R0 cos^4(lambda) / r_RE^2
```

where `lambda` is magnetic latitude and `r_RE` is geocentric radius in Earth radii.

The default script performs:

1. One optional `ANALYTIC` baseline run.
2. Three `MESH` runs with progressively finer user-defined Mode3D mesh-resolution profiles.
3. Post-processing of each shell output.
4. Comparison of the numerical cutoff to the Størmer reference.
5. A convergence check from the coarsest to the finest mesh.

The default mesh sequence is:

```text
case 1: dR_Earth = 0.20 Re, dR_boundary = 1.00 Re, coarsening = LOG
case 2: dR_Earth = 0.10 Re, dR_boundary = 0.75 Re, coarsening = LOG
case 3: dR_Earth = 0.05 Re, dR_boundary = 0.50 Re, coarsening = LOG
```

The mesh resolution is passed through the new Mode3D CLI controls:

```bash
-mode3d-mesh-res-earth-re <dRe>
-mode3d-mesh-res-boundary-re <dRe>
-mode3d-mesh-coarsening <LINEAR|LOG|EXPONENTIAL|GEOMETRIC|POWER|CONSTANT>
-mode3d-mesh-exponent <p>
-mode3d-mesh-r-boundary-re <R>
```

If these CLI controls are not available in the current checkout, C5 requires the Mode3D mesh-resolution patch to be applied first.

## Basic command

Run from the directory containing the `amps` executable:

```bash
python srcEarth/test/C5/run_C5.py -np 4 -nt 16
```

The script launches AMPS with `mpirun`, creates one subdirectory per case, and writes a top-level summary.

## Useful commands

Run a quick two-level convergence check:

```bash
python srcEarth/test/C5/run_C5.py \
  --res-earth-series 0.10,0.05 \
  --res-boundary-series 0.75,0.50 \
  -np 4 -nt 16
```

Test linear instead of logarithmic coarsening:

```bash
python srcEarth/test/C5/run_C5.py \
  --coarsening LINEAR \
  --res-earth-series 0.10,0.05 \
  --res-boundary-series 0.75,0.50
```

Test the old hard-coded mesh-resolution function without the new mesh controls:

```bash
python srcEarth/test/C5/run_C5.py --use-legacy-mesh-resolution
```

Skip the analytical baseline if C1/ANALYTIC has already been run and passed:

```bash
python srcEarth/test/C5/run_C5.py --no-analytic-baseline
```

Analyze existing outputs without rerunning AMPS:

```bash
python srcEarth/test/C5/run_C5.py --skip-run --workdir test_output/C5_mode3d_mesh
```

Show the commands without running AMPS:

```bash
python srcEarth/test/C5/run_C5.py --dry-run
```

## Output files

The top-level work directory is `test_output/C5_mode3d_mesh` by default. It contains:

```text
reference_C5_stormer.csv       analytical reference generated by the harness
C5_latitude_summary.csv        per-case, per-latitude error table
C5_aggregate_summary.csv       per-case aggregate error table
C5_result.json                 complete machine-readable report
C5_mesh_convergence.png        optional plot, if matplotlib is available
```

Each AMPS case has its own subdirectory, for example:

```text
analytic_baseline/
mesh_01_re0.2_rb1_log/
mesh_02_re0.1_rb0.75_log/
mesh_03_re0.05_rb0.5_log/
```

Each case directory contains:

```text
AMPS_PARAM_C5.in               rendered input file
C5_amps.log                    full AMPS stdout/stderr log
cutoff_3d_shells*.dat          AMPS shell cutoff output
```

## PASS/FAIL logic

C5 applies three classes of checks.

### 1. Analytical baseline

Unless `--no-analytic-baseline` is used, the script first runs the same problem with:

```bash
-mode3d-field-eval ANALYTIC
```

This should reproduce the Størmer reference. If it fails, the problem is not specific to mesh interpolation; check C1 first.

Default baseline tolerances are:

```text
|lat| <= 30 deg: max mean relative error <= 0.02
|lat| >= 60 deg: max mean relative error <= 0.15
```

### 2. Finest mesh accuracy

The finest mesh case must be close to the analytical reference. Default tolerances are:

```text
|lat| <= 30 deg: RMS and max mean relative error <= 0.05
|lat| >= 60 deg: RMS and max mean relative error <= 0.25
max longitude spread / reference <= 0.10
```

High-latitude tolerances are intentionally looser because cutoff classification near low rigidity is more sensitive to penumbra, finite domain, and trace-time effects.

### 3. Coarsest-to-finest convergence

By default, the mid-latitude RMS relative error must improve by at least 5% from the coarsest to the finest mesh:

```text
improvement = (RMS_coarse - RMS_fine) / RMS_coarse >= 0.05
```

This gate can be disabled while still reporting the metric:

```bash
python srcEarth/test/C5/run_C5.py --no-convergence-gate
```


### If every C5 mesh case fails immediately

First check the AMPS logs for mesh-generation or memory failures:

```bash
grep -i "Mode3D mesh\|memory\|error\|failed" test_output/C5_mode3d_mesh/*/C5_amps.log
```

A common failure mode is accidental over-refinement of the inactive Earth interior.
The user-defined Mode3D mesh profile is intended to control the external
backtracing domain, not to refine cells deep inside the inner loss sphere.  The
AMR callback therefore keeps the legacy coarse resolution for `r < 0.98 Re` and
applies the requested radial profile only outside that interior region.

For a fast smoke test before a full convergence sweep, run a single coarse mesh
case:

```bash
python srcEarth/test/C5/run_C5.py \
  --no-analytic-baseline \
  --res-earth-series 0.20 \
  --res-boundary-series 1.00 \
  -np 1 -nt 1
```

## What a failure means

If the analytical baseline fails, C5 is not yet testing mesh interpolation cleanly. Run C1/ANALYTIC and fix the pusher, cutoff search, or trajectory classifier first.

If the analytical baseline passes but all MESH cases fail badly, likely causes include:

- the dipole field is not being correctly stored on the Mode3D mesh;
- the mesh-backed field evaluator is using an uninitialized or stale field;
- AMR cell-centered interpolation is wrong;
- ghost cells or global field materialization are incomplete;
- the mesh cache reused an incompatible old mesh;
- MPI materialization of the field is incomplete;
- interpolation or tree search is not thread safe.

If the coarsest and finest MESH cases are both inaccurate but the finest is better, the test is doing its job: the mesh path is usable but the requested resolution is not fine enough for the desired cutoff accuracy.

If MESH passes with `-np 1 -nt 1` but fails with MPI or threads, test the parallel cases separately:

```bash
python srcEarth/test/C5/run_C5.py -np 1 -nt 1
python srcEarth/test/C5/run_C5.py -np 4 -nt 1
python srcEarth/test/C5/run_C5.py -np 1 -nt 16
python srcEarth/test/C5/run_C5.py -np 4 -nt 16
```

This separates serial mesh accuracy, MPI field materialization, and thread safety.

## Relation to C1

C1 should be treated as the strict analytical dipole cutoff benchmark. Its primary run is:

```bash
python srcEarth/test/C1/run_C1.py --mode 3d --mode3d-field-eval ANALYTIC
```

C5 is the correct place to test:

```bash
-mode3d-field-eval MESH
```

A C1/ANALYTIC pass with a C5/MESH failure means the trajectory engine is probably correct, but the mesh-backed field representation or interpolation path needs attention.
